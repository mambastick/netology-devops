## Задача 1: API Gateway

Требования к решению:

- маршрутизация запросов к нужному сервису на основе конфигурации;
- проверка аутентификационной информации в запросах;
- терминация HTTPS.

### Сравнение решений

| Решение | Маршрутизация по конфигурации | Проверка аутентификации | Терминация HTTPS | Эксплуатация | Комментарий |
| --- | --- | --- | --- | --- | --- |
| NGINX | Да: `location`, `upstream`, `proxy_pass` | Да: `auth_request`, Basic Auth, интеграция с внешним auth-сервисом | Да | Простая | Хороший reverse proxy и балансировщик, но часть API Gateway-функций приходится собирать вручную. |
| Traefik | Да: routers, services, middlewares, file/docker/k8s providers | Да: Basic Auth, ForwardAuth, middleware-плагины | Да, есть ACME/Let's Encrypt | Простая в Docker/Kubernetes | Удобен для динамической инфраструктуры, но слабее как полноценная платформа управления API. |
| Kong Gateway | Да: services, routes, upstreams, declarative config или Admin API | Да: JWT, Key Auth, Basic Auth, OAuth2/OIDC через плагины | Да | Средняя | Специализированный API Gateway с большим набором готовых плагинов и управлением политиками. |
| Envoy Proxy | Да: listeners, routes, clusters, xDS | Да: JWT filter, ext_authz, RBAC | Да | Сложная | Очень гибкий и производительный data plane, но требует больше инженерной поддержки. |
| KrakenD | Да: декларативный `krakend.json` | Да: JWT validation, интеграция с identity provider | Да | Средняя | Хорош для агрегации API и BFF-сценариев, но менее универсален как edge gateway. |

### Выбор

Для основной реализации выбран **Kong Gateway**.

Kong закрывает все обязательные требования без доработки приложения: маршруты описываются через `services` и `routes`, проверка аутентификации подключается готовыми плагинами, HTTPS завершается на gateway. В отличие от чистого NGINX, не нужно вручную собирать типовые функции API Gateway из отдельных модулей и Lua-скриптов. По сравнению с Envoy Kong проще внедрять и сопровождать, потому что большинство прикладных политик уже оформлены как плагины.

Рекомендуемая схема эксплуатации:

- запуск Kong в DB-less режиме для небольших окружений и тестовых стендов;
- declarative config в Git, изменение через CI/CD;
- отдельные плагины аутентификации и rate limiting на уровне route/service;
- TLS-сертификаты на gateway, внутренний трафик к сервисам в приватной сети.

## Задача 2: Брокер сообщений

Требования к решению:

- кластеризация для обеспечения надежности;
- хранение сообщений на диске в процессе доставки;
- высокая скорость работы;
- поддержка различных форматов сообщений;
- разделение прав доступа к потокам сообщений;
- простота эксплуатации.

### Сравнение решений

| Решение | Кластеризация | Хранение на диске | Скорость | Форматы и протоколы | Разделение прав | Эксплуатация | Комментарий |
| --- | --- | --- | --- | --- | --- | --- | --- |
| RabbitMQ | Да, cluster и quorum queues | Да, persistent messages и durable queues | Высокая для очередей задач и RPC | AMQP, MQTT, STOMP; payload может быть JSON, Protobuf, Avro и др. | Пользователи, vhosts, permissions на configure/write/read | Простая | Хороший универсальный брокер для микросервисов, очередей задач и событий. |
| Apache Kafka | Да, кластер брокеров и репликация partitions | Да, commit log на диске | Очень высокая для потоковой обработки | Бинарные сообщения; JSON, Avro, Protobuf через соглашения и schema registry | ACL на topics, groups, transactional IDs | Сложная | Отличен для event streaming и больших потоков данных, но избыточен для простых очередей команд. |
| NATS JetStream | Да, JetStream clustering на RAFT | Да, file storage | Очень высокая | Сообщения как bytes; формат задает приложение | Accounts, users, permissions на subjects | Средняя | Легкий и быстрый вариант, но меньше готовых enterprise-инструментов, чем у RabbitMQ/Kafka. |
| ActiveMQ Artemis | Да | Да, journal persistence | Высокая | AMQP, MQTT, STOMP, OpenWire, Core, JMS | Security settings, roles, permissions | Средняя | Функциональный брокер, удобен для JMS/enterprise-сценариев, но менее популярен в новых cloud-native проектах. |
| Redis Streams | Ограниченно через Redis Cluster/Sentinel | Да, через RDB/AOF | Очень высокая | Структуры Redis, payload задает приложение | Redis ACL | Простая | Подходит для легких сценариев, но по надежности и семантике доставки слабее специализированных брокеров. |

### Выбор

Для системы выбран **RabbitMQ**.

RabbitMQ лучше всего соответствует набору требований как универсальный брокер сообщений для микросервисной архитектуры. Он поддерживает кластеризацию и quorum queues для отказоустойчивости, persistent messages для сохранения сообщений на диске, гибкую модель маршрутизации через exchanges и queues, а также разграничение доступа через vhosts и permissions.

Kafka рациональнее выбирать, если основная задача - потоковая обработка больших объемов событий, аналитика и долговременный event log. В данном случае требования больше похожи на надежный брокер для межсервисного обмена, фоновых задач и событий домена. Поэтому RabbitMQ даст более простую эксплуатацию и достаточную производительность без лишней сложности.

Рекомендуемая схема эксплуатации:

- кластер RabbitMQ из трех нод;
- quorum queues для критичных очередей;
- durable queues и persistent messages для важных сообщений;
- отдельный vhost на окружение или домен;
- отдельные пользователи сервисов с минимальными permissions;
- включенный management plugin для мониторинга и диагностики.
