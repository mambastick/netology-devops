## Задача 1: Обеспечить разработку

Для организации разработки предлагается использовать **GitLab.com + GitLab CI/CD + GitLab Runner + GitLab Container Registry**.

### Схема работы

- Каждый микросервис хранится в отдельном Git-репозитории GitLab.
- В каждом репозитории находится собственный `.gitlab-ci.yml`.
- Общие шаблоны pipeline хранятся в отдельном репозитории `ci-templates` и подключаются через `include`.
- При push, merge request или tag GitLab автоматически запускает pipeline.
- Ручной запуск выполняется через `Run pipeline` с указанием переменных: окружение, версия, тип сборки, необходимость деплоя.
- Сборки выполняются на GitLab Runner: часть runner используется из GitLab SaaS, критичные runner разворачиваются на собственных серверах.
- Docker-образы сервисов публикуются в GitLab Container Registry.
- Секреты хранятся в GitLab CI/CD Variables как masked/protected variables, для production можно подключить внешний Vault.

### Соответствие требованиям

| Требование | Реализация |
| --- | --- |
| Облачная система | GitLab.com используется как SaaS. |
| Git | GitLab предоставляет Git-репозитории. |
| Репозиторий на каждый сервис | Для каждого сервиса создается отдельный project. |
| Запуск сборки по событию VCS | Pipeline запускается по push, merge request, tag. |
| Запуск сборки по кнопке с параметрами | `Run pipeline` и ручные jobs с переменными. |
| Настройки для каждой сборки | Переменные, rules, environments, artifacts, cache в `.gitlab-ci.yml`. |
| Шаблоны конфигураций | Общий репозиторий шаблонов и `include`. |
| Безопасное хранение секретов | Masked/protected CI/CD variables, file variables, интеграция с Vault. |
| Несколько конфигураций из одного репозитория | Разные jobs/stages/rules для test, build, deploy, release. |
| Кастомные шаги | Любые shell-команды в `script`, `before_script`, `after_script`. |
| Собственные Docker-образы для сборки | У каждого job можно указать свой `image`. |
| Агенты на собственных серверах | Self-hosted GitLab Runner. |
| Параллельный запуск сборок | Jobs в одной stage выполняются параллельно, runner можно масштабировать. |
| Параллельный запуск тестов | `parallel`, `parallel:matrix`, разделение тестов по jobs. |

### Обоснование

GitLab закрывает весь цикл разработки в одном продукте: хранение кода, CI/CD, registry, управление переменными, ручные и автоматические pipeline. Для микросервисов это удобно, потому что каждый сервис живет независимо, но общие правила сборки и деплоя можно централизовать через шаблоны.

Альтернативой может быть связка GitHub + GitHub Actions + GitHub Packages, но GitLab удобнее для этого сценария за счет встроенной модели runner, registry, шаблонов pipeline и единого управления CI/CD-переменными на уровне project/group.

## Задача 2: Логи

Для сбора и анализа логов предлагается использовать **Vector + Apache Kafka + Elasticsearch + Kibana**.

### Схема работы

- Приложения пишут логи в `stdout` и `stderr`.
- На каждом хосте запущен Vector-agent.
- Vector читает container logs из Docker или Kubernetes runtime.
- На стороне агента включается disk buffer, чтобы переживать временную недоступность сети или центральных сервисов.
- Vector отправляет события в Kafka с подтверждением записи.
- Vector-aggregator или Logstash читает события из Kafka, нормализует поля и записывает их в Elasticsearch.
- Kibana используется разработчиками для поиска, фильтрации, сохраненных Discover-сессий и ссылок на сохраненные поиски.

### Соответствие требованиям

| Требование | Реализация |
| --- | --- |
| Сбор логов со всех хостов | Vector-agent устанавливается на каждый хост или как DaemonSet в Kubernetes. |
| Минимальные требования к приложениям | Приложения пишут только в `stdout`/`stderr`. |
| Гарантированная доставка | Disk buffer в Vector и Kafka как долговременный буфер между агентами и Elasticsearch. |
| Поиск и фильтрация | Elasticsearch индексирует логи, Kibana дает поиск по полям и full-text. |
| UI для разработчиков | Kibana с разграничением прав доступа. |
| Ссылка на сохраненный поиск | В Kibana сохраняются Discover-сессии, ими можно делиться ссылкой. |

### Обоснование

Vector выбран как легкий агент для хостов: он умеет читать логи контейнеров, буферизовать события на диске и отправлять их дальше без изменений в приложениях. Kafka добавляется между агентами и Elasticsearch, чтобы центральное хранилище можно было обслуживать или масштабировать без потери логов. Elasticsearch и Kibana дают привычный интерфейс поиска, фильтрации и сохранения поисковых представлений.

Для небольшого стенда Kafka можно убрать и отправлять логи из Vector сразу в Elasticsearch, но для крупной микросервисной системы надежнее оставить Kafka как центральный durable buffer.

## Задача 3: Мониторинг

Для мониторинга предлагается использовать **Prometheus + Alertmanager + Grafana + node_exporter + cAdvisor + application metrics**.

### Схема работы

- На каждом хосте запускается `node_exporter` для метрик CPU, RAM, filesystem и network.
- Для контейнеров используется cAdvisor или метрики kubelet/cAdvisor в Kubernetes.
- Каждый сервис публикует собственные бизнес- и технические метрики в формате Prometheus на `/metrics`.
- Prometheus собирает метрики через service discovery или статические scrape-конфигурации.
- Alertmanager получает alert rules из Prometheus и отправляет уведомления в Slack, Telegram, email или дежурную систему.
- Grafana подключается к Prometheus как datasource и предоставляет dashboards для команд разработки и эксплуатации.

### Соответствие требованиям

| Требование | Реализация |
| --- | --- |
| Сбор метрик со всех хостов | `node_exporter` на каждом хосте, discovery targets в Prometheus. |
| Метрики CPU, RAM, HDD, Network | `node_exporter` собирает системные метрики Linux. |
| Метрики ресурсов каждого сервиса | cAdvisor/kubelet собирает CPU, RAM, network, filesystem по контейнерам. |
| Специфичные метрики сервисов | Приложения экспортируют `/metrics` в формате Prometheus. |
| UI с запросами и агрегацией | Prometheus UI и PromQL, основной интерфейс для команд - Grafana. |
| UI с панелями состояния системы | Grafana dashboards, folders, variables, permissions. |

### Обоснование

Prometheus хорошо подходит для микросервисов, потому что использует pull-модель, service discovery и единый формат метрик. Это упрощает добавление новых сервисов: достаточно опубликовать `/metrics` и добавить правила discovery. Grafana используется как основной интерфейс визуализации, потому что позволяет строить dashboards с переменными, панелями и правами доступа для разных команд.

Alertmanager добавляется для полноценной эксплуатации: мониторинг должен не только показывать состояние, но и отправлять уведомления при деградации сервисов или инфраструктуры.
